### Gamification Variant: Engagement-First Specification

**Status:** Active | **Target Platform:** React Native (iOS / Android) | **Focus:** Gamified Budgeting

This specification defines the strict visual, architectural, and behavioral rules required to shift our existing React Native platform from a *utility-first* tool to an *engagement-first* experience. All new features and component refactors must adhere strictly to these principles.

* * *

### 1\. Core Psychological Strategy

We split our interface architecture into two distinct psychological layers based on the **Octalysis Framework**. Every screen must explicitly balance these two states:

### White Hat UI (Empowerment & Growth)

*   **Purpose:** Celebrates sound financial behavior, wealth generation, and consistency.
*   **UI Rules:** Use wide layouts, smooth progress arcs, large milestone modals, and high-saturation celebratory overlays.
*   **Application:** Savings goals, hitting budget limits, logging transitions.

### Black Hat UI (Urgency & Loss Avoidance)

*   **Purpose:** Triggers immediate cognitive action when budgets are critically threatened, without inducing tracking anxiety.
*   **UI Rules:** High-contrast pulsing elements, structural countdown labels, and visibility of decaying streak blocks.
*   **Application:** Approaching category limits, unlogged daily entries, streak expiration alerts.

* * *

### 3\. Structural Code Component Standards

To maintain a cohesive tactile feel across our React Native application, you must enforce the following architecture rules on these key primitive wrappers:

### A. The Tactile Button Primitive

*   **Rule:** Standard opacity changes (`TouchableOpacity`) are strictly forbidden for primary action buttons.
*   **Requirement:** Buttons must feature a physical simulated depth using a solid bottom layer. On press, the top layer must actively offset along the Y-axis via a native spring layout.

### B. The Budget "Shield" Progress Bar

*   **Rule:** Financial data must not be displayed using flat, static linear indicators.
*   **Requirement:** Categories act as structural health shields. If spending is below 80%, use a standard static layout filled with `financialSurplus`. The second a category breaks past 80%, the component must dynamically switch to `financialDeficit` and immediately initiate a persistent opacity fade-loop to demand defensive user attention.

* * *

### 4\. Mandatory Tactile Design Principles

To ensure our application delivers a highly polished "game feel" rather than acting as a dry accounting spreadsheet, every developer must implement the following guidelines:

1.  **The "Juice" Requirement:** Every point increase or reward claim must be accompanied by visual feedback. Use lightweight canvas configurations or custom animated node collections to deploy dynamic scatter animations across the coordinate layout.
2.  **Physical Touch Validation:** Never let a significant event happen silently. Hook directly into `expo-haptics` or `react-native-haptic-feedback`. Deliver a subtle tap on entry creation, a moderate snap on button engagement, and an assertive triple-pulse sequence whenever a major saving vault is successfully cleared.
3.  **The Streak Buffer Strategy:** Never punish a user harshly for real-world volatility. Budgeting apps fail when users abandon them out of shame from breaking a streak. Build an XP-purchasable "Streak Shield" token into the backend workflow to allow active retention recovery.

* * *

### 5\. UI Layout Audit Checklist

Before launching any new screens or feature flows into production, verify they pass these engagement criteria:

*   Is the primary call-to-action utilizing a physical, pressable design rather than a flat hyperlink?
*   Does the screen canvas maintain deep contrast against primary colors to ensure rewards pop visually?
*   Are behavioral indicators (like current active streaks or profile XP) highly visible at the initial layout layer?
*   Does the interface refrain from shaming the user during financial deficit states, offering actionable escape loops instead?

### 2. Mandatory Structural Rules For All Pages
To fix the anti-patterns listed above, you must process current and future screens against these execution directives:

### Rule A: User Intent Alignment (No Static Entry Forms)

- **The Intent Split:** 80% of user sessions are for check-ins (Status Tracking); only 20% are for active entry.
- **Execution:** You must never render complex input forms as permanent, top-level layout components. Input forms must be abstracted behind an animated action overlay (e.g., a Bottom Sheet Modal or a dedicated input page) summoned by a permanent Floating Action Button (FAB).

### Rule B: Enforce Tactile Component Primitives

- **Execution:** Stop importing standard flat `PillButton` wrappers for primary actions. You must use the `TactileButton` structure. Buttons require a solid base drop-shadow layer where the button surface actively transitions downwards along the Y-axis using an elastic spring constant on press.

### Rule C: Black Hat UI Integration (Deficit Defenses)

- **Execution:** If a system calculation evaluates to an urgent state (such as `snapshot.isRedZone` or a depleted category shield), do not simply pass a red string value to a label. You must wrap the parent card component in a native hardware-accelerated looping pulse animation using `withRepeat` and `withSequence` to establish a sense of visual urgency.
